Architecture

v0.3.2 — last updated June 2026

How it's built

TalkToMeBaby is a VS Code extension split across two isolated runtimes: the extension host and the webview. The two sides communicate exclusively by message passing — they share no memory and have no direct function calls between them.

The extension host owns VS Code APIs: it reads the active document, manages synthesis (queuing audio chunks, calling TTS providers, maintaining the disk cache), persists state across reloads, and wires commands to keyboard shortcuts. The webview owns everything the user sees and hears: it renders word spans, drives playback through the Web Audio API, and streams position and state messages back to the host so the status bar and editor decorations can stay in sync.

The document model as single source of truth

When a read session starts, the extension host builds a DocumentModel from the raw document text. The model is built once and never mutated — all offsets (word positions, sentence ranges) are bounding boxes into the original text. The Chunker derives audio chunks from the model deterministically, using the same algorithm on both the host and the webview. Because both sides run the same chunker against the same model, they always agree on chunk boundaries without any round-trip negotiation.

The serialized model is posted to the webview at session start; the webview uses it to build word spans in the DOM and to drive the playback engine. The host never needs to re-derive positions from the DOM — the model is the authority.

Source layout

src/
  extension.ts          # Entry point — wires all subsystems; handles commands + resume-after-reload
  core/
    document-model.ts   # Parses raw text → DocumentModel (sentences → words with source bounding boxes)
    chunker.ts          # Groups sentences into audio chunks; deterministic on both host and webview
    timing.ts           # Merges word-boundary timings from providers into chunk-level offsets
  synthesis/
    provider.ts         # TtsProvider interface
    provider-catalog.ts # availableProviders(platform) — filters say on non-darwin
    edge.ts             # Edge TTS (msedge-tts, unofficial endpoint, exact word timing)
    elevenlabs.ts       # ElevenLabs API (exact char-alignment timing)
    say.ts              # macOS say (estimated timing)
    sarvam.ts           # Sarvam AI (estimated timing)
    synthesis-service.ts# Priority queue + dual-audio handoff; calls provider, applies cache
    disk-cache.ts       # LRU disk cache (globalStorageUri/audio-cache, default 200 MB)
    voice-cache.ts      # In-memory session cache for voice lists (never caches fallback lists)
    api-key-manager.ts  # SecretStorage read/write for ElevenLabs + Sarvam keys
    with-timeout.ts     # Race a promise against a timeout, return fallback value
  ui/
    reader-panel.ts     # ReaderPanel — webview lifecycle, message routing, post() buffer
    editor-sync.ts      # Editor decorations (sentence band) + click-to-jump listeners
    status-bar.ts       # Status bar item (sentence N/total, speed)
  webview/
    main.ts             # Webview entry — Engine, PlayerBar, SettingsPanel, Renderer
    engine.ts           # Playback engine: chunk queue, dual AudioBuffer handoff, word sweep
    renderer.ts         # DOM renderer — builds word spans from DocumentModel
    player-bar.ts       # Speed presets, slider, pause/resume, stop, gear button
    settings-panel.ts   # Settings slide-in: Provider / Voice / Appearance sections

Tests live alongside the files they test (*.test.ts co-located in src/). Vitest runs them in a happy-dom environment — no VS Code runtime is needed to run the test suite.

The five key invariants

These rules govern every structural decision in the codebase. If a change would break any of them, reconsider the change.

1. Document model is the single source of truth

DocumentModel is built once from raw text and never mutated. All offsets (word positions, sentence ranges) are bounding boxes into the original text. The chunker derives chunks deterministically from the model, so the host and the webview always agree on chunk boundaries.

2. Webview owns playback state

The webview engine drives position and state messages; the extension host updates the status bar and persists position in response. The host never assumes playback state — it reads it from messages.

3. Reconfigure in place; never auto-play

Provider and voice switches call ReadingSession.reconfigure(). That method aborts the old synthesis pipeline, builds a new one, and re-initialises the same webview paused at the current sentence. The user presses play when ready. There is no surprise audio after a settings change.

4. Never cache fallback voice lists

VoiceCache stores only real provider voice lists (length > 1). If listVoices() times out, the fallback — a list containing only the default voice — is returned for display but not stored. This ensures the next fetch retries the real list. Caching the fallback would pin the provider to a single voice for the whole session.

5. Disk cache uses globalStorageUri

DiskCache writes synthesised audio to context.globalStorageUri/audio-cache. The size limit is controlled by the talktomebaby.cacheSizeMB setting (default 200 MB). Eviction is LRU, triggered on write.

Adding a TTS provider

The synthesis layer is provider-agnostic. Adding a new voice engine takes five steps:

  1. Create src/synthesis/your-provider.ts implementing TtsProvider (see src/synthesis/provider.ts for the interface).
  2. Add an entry to src/synthesis/provider-catalog.ts — supply id, label, description, requiresKey, and defaultVoice.
  3. Register the provider in extension.ts inside the makeProviderById() switch.
  4. Add the config key talktomebaby.voice.your-provider to package.json.
  5. If the provider requires an API key, add handling in src/ui/api-key-manager.ts.

For a worked example, compare edge.ts (exact word timing) with say.ts (estimated timing). See Providers for end-user documentation on the built-in providers.

Contributing

Dev environment

Fork and clone the repo, run npm install, open the project in VS Code, and press F5 to launch the Extension Development Host. VS Code runs npm: watch as the default build task, so the extension rebuilds incrementally as you save. Reload the dev host window to pick up changes.

Running the tests

The test suite runs with Vitest in a happy-dom environment — no VS Code runtime is needed: 

npm run compile   # dev build: esbuild ext (→ out/extension.js) + webview (→ media/reader.js)
npm test          # vitest run — no VS Code runtime needed

Tests are co-located with the source files they cover (*.test.ts inside src/).

PR expectations

  • Open an issue first to discuss non-trivial changes.
  • Keep PRs focused — one feature or fix per PR.
  • Add *.test.ts tests for any new core or synthesis logic (the test suite runs without a VS Code runtime).
  • No API keys in code — keys go through VS Code SecretStorage via ApiKeyManager.
  • Run npm run compile and npm test — both must pass.
  • Update CHANGELOG.md if adding features or fixing bugs.

For bug reports, use the GitHub issue template and include your VS Code version, OS, extension version, and the relevant output from the Output panel > "TalkToMeBaby".
