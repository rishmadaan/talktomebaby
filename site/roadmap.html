Roadmap

Last updated v0.3.2 — 2026-06-10

Where things stand

v0.3.2 made a meaningful change to the default provider story: macOS now defaults to the official say provider — the Apple speech synthesis CLI that ships with every Mac. It's fully offline, no API key required, and no data leaves your machine. That removes one long-standing rough edge for macOS users.

Windows and Linux still default to Edge TTS — free, no key needed, and neural quality. But Edge TTS is built on an unofficial Microsoft endpoint (the same one Edge's "Read Aloud" feature uses, via the msedge-tts package). Microsoft offers no contract for it; it has already changed once (a migration from package v1 to v2 was required) and could break or get gated at any time. That is the price of free.

Two structural gaps remain.

Structural gaps

Unofficial default on Windows & Linux

macOS defaults to the official say provider. Windows and Linux still default to Edge TTS — unofficial, best-effort. If the Edge endpoint breaks upstream, those users lose audio until a fix ships or they configure a key-based provider. Mitigation today: three alternative providers (ElevenLabs, Sarvam, and say on macOS). Real mitigation: gap 2.

Offline reading is macOS-only

The only provider that works without network access and without an API key is macOS say. Windows and Linux users have no offline or no-key option except Edge TTS — which requires network anyway. There is currently no path to offline reading on those platforms.

The plan: bundled local neural TTS

The obvious fix would be per-OS system voices: Windows SAPI via PowerShell and Linux espeak-ng / speech-dispatcher alongside macOS say. That means three separate integrations, three different quality levels (espeak-ng in particular sounds robotic), three maintenance surfaces — and still only estimated word timing on all of them.

The better path is one bundled local neural TTS engine that runs identically on every OS. Two candidates from the ONNX ecosystem:

Candidates

  • Piper — small (~50–100 MB per voice), fast on CPU, MIT-licensed, good quality, runs via onnxruntime. The primary candidate and strongest fit for this use case.
  • Kokoro — higher quality (~82M params), also ONNX-runnable; heavier than Piper.

What it buys

  • A true offline default that works the same on macOS, Windows, and Linux.
  • No API keys and no unofficial endpoints in the default path.
  • Phoneme timestamps from the engine itself — better-than-estimated word highlighting, closer to the exact timing ElevenLabs and Edge TTS provide.

What it costs

  • Model download flow. Bundling a voice model in the VSIX would bloat the install. The likely design is a one-time "download voice (~60 MB)" prompt on first use of the local provider.
  • onnxruntime-node as a native dependency. Native modules in VS Code extensions require careful packaging and platform-specific binaries — non-trivial engineering work.

Sequencing

macOS already uses say as the zero-config default as of v0.3.2. The next step is the Piper provider for Windows and Linux: when Piper lands, it becomes the offline default on those platforms — closing both structural gaps in a single move.

The macOS default stays as say until Piper quality is confirmed comparable or better. If it is, Piper can become the single default across all three platforms, unifying the experience entirely.

Current default summary: macOS → say (official, offline). Windows & Linux → Edge TTS (unofficial, requires network). The roadmap replaces Edge TTS as the Windows/Linux default with a Piper-based local provider.

See Providers for detail on the current provider landscape, including fragility notes and how API keys are stored. See Architecture for how providers plug into the synthesis pipeline.
